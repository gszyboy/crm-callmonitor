# 鸭暖屯泉 CRM 来电弹屏 APK - PRD

> 模块名：crm-callmonitor
> 版本：v0.9
> 日期：2026-06-26
> 状态：待定稿
> **本 PRD 只描述 APK 自己做什么。APK 与后端的接口约定见 `../AGENTS.md` §7。后端需要做什么见 `BACKEND-REQUIREMENTS.md`。**

---

## 1. APK 做什么（一句话版）

APK 装在中心调度员手机上，**只做一件事**：**当调度员手机接到来电时，把"来电号码 + 调度员标识"通过 HTTP POST 推给后端**。其他什么都不做。

---

## 2. APK 用到的后端接口

APK 调后端的 2 个接口（**接口的详细约定**见 `../AGENTS.md` §7）：

| # | API | 何时调用 |
|---|---|---|
| 1 | `POST /api/call/bind-confirm` | 首次配置（保存服务器地址 + 绑定码时）|
| 2 | `POST /api/call/incoming` | 每次来电响铃时 |

APK 不会调用任何其他接口。

---

## 3. 用户角色

| 角色 | APK 视角 |
|---|---|
| **中心调度员**（`admin_user.role='dispatcher'`）| 唯一使用者。APK 装在他手机上 |
| 其他角色 | 不涉及 |

**APK 端不假设客户公司调度员数量**（1 个、2 个、10 个都可以；APK 行为完全相同——每台 APK 装在一个调度员的手机上，独立工作）。

## 4. APK 工作流程

### 4.1 首次使用（一次性配置）

#### 4.1.1 方式 A：手动填写表单

```
[1] 调度员用手机扫码下载 APK
       ↓
[2] 调度员打开 APK，看到表单
       ┌──────────────────────────────┐
       │  📞 来电推送助手                │
       │                              │
       │  [📷 扫一扫]  ← 按钮          │
       │                              │
       │  服务器地址：                  │
       │  [ https://crm.example.com ]  │
       │                              │
       │  调度员标识：                  │
       │  [           123           ]  │
       │                              │
       │  绑定码：                     │
       │  [        A8X9K2          ]  │
       │                              │
       │       [ 保存并启动 ]          │
       └──────────────────────────────┘
       ↓
[3] 调度员点"保存并启动"
       ↓
[4] APK 调用 bind-confirm 接口（接口约定见 AGENTS §7）
       ↓ 成功
[5] APK 收到 api_token，加密存本地
       ↓
[6] APK 启动前台服务，状态栏出现"📞 来电推送助手 运行中"
       ↓
[7] 调度员可关闭 APP，APK 在后台常驻
```

#### 4.1.2 方式 B：扫描二维码

```
[1] 调度员在 PC 端管理后台 → 调度员管理 → 点"生成绑定码"（PC 端展示二维码）
       ↓
[2] 调度员用手机打开 APK
       ↓
[3] 调度员点"📷 扫一扫"按钮
       ↓
[4] APK 检查 CAMERA 权限
       ├─ 未授权 → 弹权限申请对话框
       └─ 已授权 → 直接跳第 5 步
       ↓
[5] 调起相机扫码
       ↓
[6] 解析二维码 → 自动回填 3 个字段（服务器地址、dispatcher_id、code）
       ↓
[7] 调度员看到 3 个字段已被填充，**点"保存并启动"**
       ↓
[8] 走 §4.1.1 步骤 [4] 及之后的所有步骤（bind-confirm → 存 token → 启动前台服务）
```

**二维码内容格式**（**这是给 PC 端看的**，APK 只负责解析）：

```json
{
  "server_url": "https://crm.xiyutianhe.com",
  "dispatcher_id": 123,
  "code": "A8X9K2",
  "ts": 1719400000
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `server_url` | string | ✅ | 后端完整 URL（含 https://）|
| `dispatcher_id` | int | ✅ | 当前登录的调度员 ID |
| `code` | string | ✅ | 6 位绑定码（10 分钟内有效）|
| `ts` | int | ✅ | 二维码生成时间戳（用于可选的过期提示）|

**APK 行为**：
- 解析后**只回填表单**，**不直接提交**（给用户复核的机会）
- 4 个字段任何一个缺失 → 提示"二维码格式错误"
- `ts` 字段：**APK 不强制校验过期**。如果用户扫码时二维码已生成超过 10 分钟，APK **仅在 UI 提示**"二维码可能已过期，请让管理员重新生成"，但**仍允许回填**（最终由绑定接口返回的错误码决定是否生效）

> **为什么不做强制校验**：APK 时钟可能不准（用户手动调过时间），强校验会误判。过期由后端绑定接口判断最准。

### 4.2 日常使用（每次来电）

```
[1] 客户拨打调度员手机
       ↓ 100ms 内
[2] Android 系统派发 PHONE_STATE_CHANGED 广播
       ↓
[3] APK 监听到广播，读取来电号码
       ↓
[4] APK 检查 DedupStore：此号码 10 秒内**已成功推送**或**已在重试队列**？
       ├─ 是 → 忽略（不重复推）
       └─ 否 → 继续
       ↓
[5] APK 提交到来电事件队列（ExecutorService 单线程池）
       ↓
[6] 队列消费线程调 incoming 接口：
       POST {SERVER_URL}/api/call/incoming
       Query: ?caller=138xxxx
              &dispatcher_id=123
              &device_id=本机唯一ID
              &token=api_token
              &ts=当前时间戳
       ↓ 200ms 内完成
[7] APK 收到 200 响应后，**对用户什么都不做**（不弹窗、不提示），
       但**对内部**：① 记入 DedupStore（防止重推）② 如在 RetryQueue 中则删除该条
       ↓
       失败（5xx/超时/网络断）→ 入 RetryQueue，等网络恢复重试
```

### 4.3 失败重试

| 失败类型 | 行为 |
|---|---|
| **网络错误**（超时/连接失败/5xx）| **入本地重试队列**，下次网络可用时重试（最多 3 次）|
| **业务错误**（403/404）| **不入队列**，立即更新状态（403 → 通知 MainActivity 显示"已解绑"；404 → 显示"账号异常"）|
| **成功推送** | 从队列中删除（如果存在）；记入"已推送"集合（防止重试和重推）|

**去重规则**：用 `Map<号码, 时间戳>` 记录"已成功推送 OR 已在重试队列"的号码，10 秒内不重复入队。

### 4.4 设备更换

```
[1] 调度员在管理端被解绑（后端流程，APK 不管）
[2] 旧手机 APK 下次推来电时收到 403
[3] 旧手机 APK 检测到 403 响应后**立即清掉本地的 api_token 和 dispatcher_id**
[4] 旧手机 APK 提示"已解绑，请联系管理员重新绑定"
[5] 调度员重走 §4.1 首次使用流程
```

## 6. 异常场景与边界处理

> **本章是 PRD 核心**：程序员只读 §4 正常流程，写出来的代码**会在国产机上一败涂地**。本章明确所有异常场景的产品行为。

### 6.1 国产机后台管理（最关键，决定 APK 能不能用）

| 场景 | 用户感受 | APK 行为 |
|---|---|---|
| **首次启动后** | 用户看到引导弹窗 | 弹窗强制出现，文案："为确保来电能正常推送，请允许 APP 在后台运行" + 3 张操作截图（电池优化/自启动/锁屏保持运行）|
| **用户拒绝授权 / 不加白名单** | APK 仍能启动 | 第一次来电来时**再次弹引导**（不是无限弹，每天最多 1 次）|
| **国产机系统更新后** | 用户的白名单可能被清 | 启动时检测白名单状态，被清了再次引导 |
| **华为/小米/OPPO/Vivo 跳转失败** | 用户看到通用电池优化页 | 5 大厂商 componentName 全部 try-catch，失败回退到通用入口 |

**关键权限申请优先级**（**用户拒绝后影响**）：
- `READ_PHONE_STATE`：**必须给**。不给则核心功能不可用，每次启动都申请。
- `POST_NOTIFICATIONS`：可不给。状态栏通知不显示，功能不受影响。
- `CAMERA`：可不给。用户只能用手动填写方式（§4.1.1）。

### 6.2 Doze 模式（Android 6+ 深度休眠）

| 场景 | APK 行为 |
|---|---|
| 手机进入 Doze（屏幕关闭长时间不动）| `PHONE_STATE_CHANGED` 广播**可能不响应** |
| 唤醒后（用户操作手机）| 系统批量补发未送达的广播，APK **能正常收到** |
| **应对** | **首次启动必须引导用户把 APP 加入"电池优化白名单"**（不加入的话 Doze 下 APK 收不到任何广播，**所有来电都丢失**）|

### 6.3 来电状态机处理（核心，必须严格区分）

Android `PHONE_STATE_CHANGED` 广播的状态机：

```
IDLE（待机）→ RINGING（响铃）→ OFFHOOK（摘机/通话中）→ IDLE（挂断）
```

| 状态变化 | 含义 | APK 行为 |
|---|---|---|
| `IDLE → RINGING` | **来电响铃** | **✅ 推送**（取 `EXTRA_INCOMING_NUMBER`）|
| `IDLE → OFFHOOK` | **主动去电**（调度员打出）| **❌ 不推送**（不是来电）|
| `RINGING → OFFHOOK` | 调度员接听 | **❌ 不推送**（已处理过）|
| `RINGING → IDLE` | 客户挂断/未接 | **❌ 不推送**（已处理过 RINGING）|
| `OFFHOOK → IDLE` | 通话结束 | **❌ 不推送**（不是新事件）|
| `OFFHOOK → RINGING`（再次）| **呼叫等待**（通话中又来一个）| **✅ 推送**（这是新来电，号码是**新来电方**）|

**实现要点**：
- 只在 `EXTRA_STATE_RINGING` 状态时处理
- 用 `state == RINGING && previousState != OFFHOOK` 判断"是新来电"（防止 OFFHOOK → RINGING 被误判为去电）

### 6.4 来电号码异常

| 场景 | APK 行为 |
|---|---|
| `EXTRA_INCOMING_NUMBER` 为 `null` | **❌ 不推送**（不调 HTTP）|
| 号码为空字符串 | **❌ 不推送** |
| 号码为 `+86` 开头（标准国际格式）| 标准化为 `138xxxx`（去 `+86` 和前导 0）后推送 |
| 号码为 `0` 开头（带长途区号）| 标准化为标准格式后推送 |
| 号码为 `+` 开头的国际号码 | **❌ 不推送**（本期不支持国外来电，不污染后端数据）|
| 号码为 `*` 或 `#` 开头（**测试号码/特服号**）| **❌ 不推送**（明显不是真实来电）|
| 双卡手机副卡来电 | **❌ 不推送**（Android 系统拿不到副卡号码，推送会污染数据）|
| 调度员拒接/未接 | **✅ 仍推送**（客户档案已弹给调度员看，业务流程完成）|

**号码标准化函数**（伪代码）：
```
function normalize(rawNumber):
    if rawNumber is null or empty: return null
    if rawNumber 包含 * 或 #: return null
    if not rawNumber 以 + 开头: return null  // 本期只支持国内号
    number = rawNumber.replace("+86", "").replace(/^0/, "")
    if number 不是 11 位数字: return null
    return number
```

### 6.5 边界场景

| 场景 | APK 行为 |
|---|---|
| **用户首次启动就拒绝所有权限** | APP 仍能打开表单界面，但 bind-confirm 会失败（"必给权限"），提示"必须授予" |
| **APK 已在运行，用户卸载重装** | 视为新用户，重走 §4.1 首次配置流程 |
| **APK 升级（v0.6 → v0.7）** | 本地 SharedPreferences 数据**不丢**（key 命名加版本前缀）；api_token 仍有效 |
| **APK 长时间未启动（30 天）** | api_token 可能已被后端失效，bind-confirm 重新走一遍 |
| **手机没插 SIM 卡** | 来电根本进不到手机，APK 收不到广播，**正确行为，不算 bug** |
| **APK 进程被系统杀掉后 5 分钟内再次启动** | 启动时**自动**加载重试队列（如果有未完成的重试）|
| **两台 APK 在同公司不同手机** | 各自独立工作，不互相影响（每台 APK 用自己的 dispatcher_id）|
| **APK 启动时网络不可用** | 启动时调 bind-confirm 会失败，提示"网络不可用，请稍后重试" |

### 6.6 错误响应处理

| 后端响应 | 用户感受 | APK 内部 |
|---|---|---|
| **200** | 无感（不弹任何东西）| 记入 DedupStore，删除 RetryQueue 中该条 |
| **403** | 显示"已解绑，请联系管理员重新绑定" | **清掉本地 token**，弹窗引导重配 |
| **404** | 显示"调度员账号异常" | 不清 token（可能是后端临时错误），弹窗提示 |
| **5xx / 超时 / 网络断开** | 无感 | 入 RetryQueue，等网络恢复重试 |
| **业务错误 + 已入重试队列** | 无感 | 错误响应**不入重试队列**（403/404 是"明确的"业务失败，重试也白搭）|

### 6.7 安全与异常

| 场景 | 行为 |
|---|---|
| **APK 安装包被反编译** | dispatcher_id / api_token 是普通字符串，反编译可见。本期**接受这个风险**（v0.6 决策 B，不做 HMAC 签名）|
| **设备被 root** | 攻击者可提取 api_token 伪造推送。本期**接受这个风险**（v0.6 决策）|
| **APK 内存中 token 被 dump** | 同样接受风险。生产环境建议加 ProGuard 混淆字符串 |
| **dispatcher_id + api_token 在日志中泄露** | 异常日志**绝不打印 token**（AGENTS §4.5 已规定）|

## 7. 非功能需求

### 7.1 性能

| 指标 | 要求 | 测量方法 |
|---|---|---|
| 端到端延迟（来电响铃到 HTTP 推送完成）| **< 500ms**（局域网）| 用 `LogCat` 打时间戳 |
| 来电到入事件队列 | **< 50ms** | 同上 |
| APK 安装包大小 | **< 2MB** | `aapt dump badging` |
| 冷启动时间（点图标到 MainActivity 显示）| **< 1s** | 人工秒表 |
| 后台内存占用 | **< 30MB** | Android Studio Profiler |
| 待机耗电 | **< 1%/小时** | Battery Historian |

### 7.2 可靠性

| 指标 | 要求 |
|---|---|
| 来电捕获率（客户打到调度员手机时）| **≥ 99%**（前提：用户加了白名单）|
| 来电推送成功率（来电到后端）| **≥ 99%**（重试 3 次内）|
| 误推率（去电/号码空被推）| **0%**（PRD §6.3 §6.4 严格过滤）|
| 重启后重试队列恢复 | **100%**（SharedPreferences 持久化）|
| 网络断开时来电不丢失 | **100%**（入重试队列）|
| 网络恢复后重试成功 | **100%**（NetworkWatcher 触发）|

### 7.3 可用性

- 用户**首次安装**后**必须看到**白名单引导弹窗
- 用户**未加白名单**时再次来电，应**再次引导**（每天最多 1 次）
- 任何错误提示必须**中文**（不能用英文）
- 错误提示**不能包含技术术语**（如 HTTP 500、SocketTimeout）

### 7.4 兼容性

| 项 | 范围 |
|---|---|
| Android 版本 | minSDK 24（Android 7.0）~ targetSDK 34（Android 14）|
| 手机厂商 | 华为/小米/OPPO/Vivo/三星/魅族（**主流国产**全部适配）|
| 屏幕 | 任意分辨率（UI 用 dp 单位，不写死 px）|
| 系统语言 | 中文（简体）|
| 网络 | WiFi/4G/5G/3G（自动切换）|
| 进程被杀后 | BroadcastReceiver 自启、前台服务自启 |

### 7.5 可维护性

- 代码总量 **< 1000 行 Java**（不含第三方库）
- 每个 Java 类 **< 200 行**
- 方法 **< 30 行**（过长方法必须拆分）
- 注释：每个类顶部说明**职责**；每个 public 方法说明**入参/出参/副作用**

### 7.6 可测试性

- 所有"业务逻辑"（去重逻辑/重试触发/号码标准化）**纯函数化**（不依赖 Android Context），方便 JUnit 测试
- BroadcastReceiver 收到广播后**只解析参数**，把业务逻辑交给 `CallProcessor` 类（可单元测试）
- 重试队列用接口抽象（`IRetryStore`），方便 Mock

### 7.7 安全（本期接受的风险）

- ❌ 不做 HMAC 签名（v0.6 决策 B）
- ❌ 不做证书锁定
- ❌ 不做 root 检测
- ❌ 不做反调试
- ⚠️ 已知风险：root 设备 + 提取 token → 可伪造推送

## 8. 监控与埋点

> **本期不上**集中式 APM（如 Sentry），但**APK 必须埋点**以便客户反馈问题时能查日志。

### 8.1 必埋点事件（写到 LogCat，tag=`crmcallmonitor`）

| 事件 | 触发时机 | 关键参数 |
|---|---|---|
| `app_start` | APP 启动 | dispatcher_id（有/无）|
| `app_config_saved` | 用户点"保存并启动" | dispatcher_id, has_token |
| `bind_success` | bind-confirm 返回 200 | dispatcher_id, device_id |
| `bind_failed` | bind-confirm 返回非 200 | code, error_msg |
| `incoming_call` | 监听到新来电 | caller, normalized_caller, is_offhook_to_ring |
| `push_success` | incoming 返回 200 | caller, ts |
| `push_failed_403` | incoming 返回 403 | caller |
| `push_failed_404` | incoming 返回 404 | caller |
| `push_failed_network` | incoming 网络错误 | caller, retry_count |
| `push_retry_success` | 重试成功 | caller, total_retry_count |
| `push_giveup` | 重试 3 次后放弃 | caller, total_retry_count |
| `network_recovered` | NetworkWatcher 检测到网络恢复 | - |
| `queue_drained` | 重试队列清空 | - |
| `unbound_detected` | 收到 403 清理本地 | - |
| `app_killed` | APP 进程被系统杀掉（onTaskRemoved）| - |

### 8.2 关键指标（运维统计，本地不存，仅打日志）

- 今日来电量（按 dispatcher_id 分组）
- 弹窗命中率（成功推送 / 总推送尝试）
- 失败推送分类（5xx/超时/网络）
- 重试队列平均大小

### 8.3 异常上报策略

- ❌ **不上报**到远端（避免流量和隐私问题）
- ✅ **写日志到本地**（`/sdcard/Android/data/<package>/logs/` 目录，每天一个文件）
- ✅ 用户反馈问题时，**引导用户把日志目录打包发给客服**
